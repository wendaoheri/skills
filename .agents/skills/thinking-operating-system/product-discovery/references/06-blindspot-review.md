# 盲点挑战与反方评审参考

本文件提供用于产品盲点挑战的框架和模板，帮助团队从反方视角识别隐含假设、潜在风险以及可能失败的原因。在产品发现流程的盲点挑战阶段，可参照此文件撰写评审意见。

## 常见盲点清单

以下是产品设计过程中常见的盲点示例，可结合具体场景进行扩展或删减：

1. **目标用户不明确**：产品针对人群过于广泛或模糊，无法洞察真实需求。
2. **问题强度不足**：痛点不够严重或频率低，用户不愿改变现有习惯或投入成本。
3. **场景过度泛化**：试图覆盖多个场景，导致重点不突出，产品边界模糊。
4. **过度依赖模型或技术**：以为先进技术就是壁垒，却忽视使用场景和数据支撑。
5. **缺乏数据或能力壁垒**：核心能力易被复制，或缺乏闭环数据，难以持续优化。
6. **分发渠道模糊**：产品难以触达用户，缺乏明确获客和推广路径。
7. **商业模式缺失**：没有清晰的付费场景或商业价值支撑，或不符合企业战略。
8. **执行成本被低估**：忽略技术、数据、运营、合规、安全等实施成本。
9. **合规与安全风险**：数据隐私、行业监管、算法伦理等未充分评估。
10. **用户改变行为难度大**：用户需要改变现有工作流程或工具，迁移成本高。

## 反方挑战模板

撰写盲点挑战时，可以参考以下结构：

```
### 最强反方观点

从最苛刻的角度阐述为什么这个产品可能没有价值或难以成功，例如：
- 痛点并不强，现有方案已足够好；
- 竞争对手拥有更好的资源和渠道；
- 技术并不具备独特性，容易被复制；
- 用户体验复杂，使用门槛高；
- 商业模式不明确，难以实现盈利。

### 最大的 5 个盲点
1. …
2. …
3. …
4. …
5. …

### 最危险的隐藏假设

列出团队可能忽视的隐含假设，例如：
- 用户愿意付费；
- 数据源稳定且可用；
- 监管允许该功能的上线；
- 技术可以顺利落地；
- 用户愿意改变现有工作流程。

### 如果产品失败，最可能的原因

分析产品失败的主要原因，例如市场不接受、技术不可行、竞争失利、商业模式无盈利点、部署运营成本过高等。

### 要让产品成立，必须证明什么

总结关键待验证的假设，指出需要通过验证计划解决的问题，例如：
- 核心用户是否真的存在且有需求？
- 问题是否足够痛？
- 模型或算法是否足以支撑关键能力？
- 用户付费意愿如何？
- 产品能否触达目标用户？
```

通过反方挑战，团队可以及早发现产品方案中的薄弱环节，制定针对性的验证计划或调整方案，从而提高产品成功的概率。
