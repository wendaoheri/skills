# 问题定义参考

本文件提供在产品发现流程中用于界定核心问题的模板、评估指标和常见伪需求示例。在阶段 1 和阶段 2 时，你可以参考这里的内容来引导用户把“想做什么”转换成“要解决什么问题”。

## 核心问题定义模板

使用以下句式帮助用户明确产品要解决的真实问题：

> 对于【目标用户】，当他们在【具体场景】下，需要完成【任务】时，现有方式存在【痛点或成本或风险】，因此需要一个能够【核心能力】的产品。

说明：

- **目标用户**：明确是谁会用或者受益，避免泛泛而谈。
- **具体场景**：描述用户在什么情况下遇到问题，越具体越好。
- **任务**：用户在场景中要完成的工作或达到的目的。
- **痛点/成本/风险**：现有方法为什么不能满足需要，是真痛点还是表面不便？
- **核心能力**：该产品提供什么关键价值来解决问题。

## 问题强度评估

评估一个问题是否值得解决，可以从以下维度打分或描述：

1. **频次**：问题出现的频率，是日常刚需还是偶发需求？
2. **刚性**：如果不解决，用户会遭受什么损失？是关键性任务还是可有可无？
3. **痛点强度**：现有方案是否真的无法满足需求，还是仅存在小幅优化空间？
4. **预算意愿**：用户是否愿意付费解决这个问题，或者在企业场景下是否有预算支持？
5. **替代方案**：是否已有成熟方案或通用工具可以解决？替代方案的效果是否已足够好？
6. **适用性**：是否适合通过软件、AI 或 Agent 解决，还是线下服务或流程优化更合适？

## 常见伪需求

在对问题进行定义时，要警惕以下常见伪需求陷阱：

- **好玩但不刚需**：用户觉得有趣，但痛点不强或不会持续使用。
- **技术驱动而非需求驱动**：产品是因为技术可行而存在，而不是实际用户需求推动。
- **解决表层问题**：产品关注了表面流程，却没有解决根本痛点，例如效率提升但决策依赖关系未解决。
- **由内部假设推导**：基于团队内部假设或领导意志，而非真实用户反馈。
- **服务更合适**：问题本质上需要咨询、执行或流程改造，而非单纯的软件产品。
- **价格敏感度被低估**：用户虽然有需求，但对付费或额外成本极其敏感，付费意愿不强。

## 输出格式示例

在阶段 2 中，结合本模板输出问题定义，可参考如下结构：

```
### 核心问题定义

对【核心用户群体】来说，当他们在【具体场景 A】想要完成【任务 B】时，现有做法【如何解决】存在【痛点 C】，因此需要一个能【核心能力 D】的产品。

### 问题强度评估

- 频次：★★★☆☆（较高频次）
- 刚性：★★★★☆（不解决会影响业务目标）
- 痛点强度：★★★☆☆（当前方案有明显痛点）
- 预算意愿：★★★☆☆（用户愿意为效率提升付费）
- 替代方案：有，但体验差；存在差异化机会
- 适用性：适合通过 AI Agent 解决

### 可能的伪需求

- 方案更多是锦上添花，而非雪中送炭
- 缺乏真实用户反馈，风险不明
```
